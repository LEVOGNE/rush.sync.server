How It All Started: One Developer, One Idea, One Language

L
LEVOGNE
Solo Developer & Founder

In February 2025, I sat in front of an empty cargo new project and asked myself a simple question: Why is it so complicated to just put a website online?

Nginx configs, Apache virtual hosts, Certbot cron jobs, Docker orchestration, reverse proxy rules — all for serving a simple HTML file. There had to be a better way.

What if you could just type rush create mysite and it's live? No config files, no DNS setup, no certificate management. Just running.

That idea became Rush Sync Server. Not a startup with a team of twenty. Not a VC-funded moonshot. Just one developer, writing Rust, solving a real problem.

I chose Rust deliberately. Not because it's trendy — because it's honest. It forces you to think about every allocation, every error path, every edge case. When you're a team of one, the compiler is your best colleague.

The first version was 200 lines. A single HTTP server serving static files. It didn't even have a name yet.

May 2025

Building the Reverse Proxy: When Subdomains Become Magic

Three months in, I had a working server manager. You could create servers, start and stop them, serve files. But every server needed its own port. localhost:8000, localhost:8001, localhost:8002. Not exactly user-friendly.

So I built a reverse proxy. The kind of thing that normally requires Nginx or Traefik or Caddy with a YAML file longer than your arm. Mine fits in a single Rust file.

The concept is simple: every server gets a subdomain. Create a server called "blog" and it's instantly reachable at blog.yourdomain.com. No DNS changes, no config reload. The wildcard DNS handles resolution, the proxy handles routing.

A request comes in for blog.rush-sync-server.com. The proxy strips the subdomain, looks up which port "blog" runs on, and forwards the request. Total overhead: <1ms.

Then came HTTPS. Auto-generated self-signed certificates for development, Let's Encrypt integration for production. You don't configure TLS — it just happens.

September 2025

From Side Project to Production: The Milestones

Building something alone means you wear every hat. Developer, architect, tester, designer, DevOps, tech writer. Some days you write beautiful algorithms. Other days you spend four hours debugging a CSS margin.

Here's the journey so far:

  • Feb 2025 Day zero. First cargo new rush-sync-server. Basic HTTP file server.
  • Mar 2025 Server Manager. Create, start, stop, restart. Multi-server support with port allocation.
  • Apr 2025 TUI Interface. Full terminal UI with themes, command history, keyboard navigation.
  • May 2025 Reverse Proxy. Subdomain-based routing. One wildcard DNS, infinite servers.
  • Jul 2025 Auto HTTPS. Self-signed certs for dev, Let's Encrypt for production. Zero config.
  • Oct 2025 WebSocket Sync. Real-time file synchronization across devices.
  • Dec 2025 Docker & Headless. Multi-stage Docker build, API-first headless mode.
  • Feb 2026 API Security. HMAC key hashing, rate limiting, timing-safe auth. Production-ready.

What's Next: The Road Ahead

Rush Sync Server is no longer a side project. It's a production system running on real hardware, serving real websites, handling real traffic. But the vision goes further.

The vision: 100+ servers with full orchestration on a single machine, with absolute perfect performance. The roadmap includes SQLite persistence for server state, PostgreSQL support for enterprise deployments, a plugin system for custom middleware, and cluster mode for horizontal scaling.

The core philosophy stays the same: simplicity. Every feature must make the developer's life easier, not harder. If it needs a config file, it's not done yet. If it needs documentation longer than a tweet, it's too complex.

I'm looking for investors and partners who believe that infrastructure software doesn't have to be complicated. If you want to be part of this journey — let's talk.

This is what happens when one person refuses to accept that "it's always been done this way." Rush Sync Server exists because I needed it. And I think you might need it too.
